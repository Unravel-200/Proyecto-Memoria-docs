PROYECTO MEMORIA
ARQUITECTURA DE PROGRAMACIÓN C++ + BLUEPRINTS
=============================================

PROPÓSITO
---------
Este documento define cómo se distribuirá la programación del juego entre C++ y Blueprints.

Debe entregarse a Codex o Claude Code cuando la tarea involucre:

- Programación.
- Creación de clases.
- Sistemas del jugador.
- IA.
- Guardado.
- Interacción.
- Eventos narrativos.
- UI.
- Arquitectura del proyecto.
- Refactorización.
- Revisión de Blueprints.
- Decisiones sobre Level Blueprint.

DECISIÓN DEFINITIVA
-------------------
[X] Utilizar C++ y Blueprints.
[X] Realizar la mayor parte de la lógica central en C++.
[X] Utilizar Blueprints como capa de contenido, configuración y presentación.
[D] Usar únicamente Blueprints.
[D] Usar únicamente C++.

PRINCIPIO GENERAL
-----------------
C++ proporciona sistemas estables, reutilizables y fáciles de revisar en Git.

Blueprints proporcionan iteración rápida para:

- Diseño.
- Eventos.
- Arte.
- Audio.
- UI.
- Animación.
- Configuración específica.

REGLAS OBLIGATORIAS
-------------------
[ ] No colocar sistemas centrales en Level Blueprint.
[ ] Level Blueprint solo se usa para casos locales muy específicos y documentados.
[ ] No duplicar lógica entre C++ y Blueprint.
[ ] No escribir en C++ secuencias narrativas rígidas que deban ajustarse con frecuencia.
[ ] Exponer a Blueprint solamente lo necesario.
[ ] Usar BlueprintReadOnly cuando Blueprint no deba modificar una propiedad.
[ ] Usar EditDefaultsOnly para valores que pertenecen a la clase.
[ ] Usar EditInstanceOnly solo cuando sea necesario configurar instancias.
[ ] Usar BlueprintCallable únicamente para funciones que Blueprint realmente necesita.
[ ] Usar BlueprintImplementableEvent para presentación configurable.
[ ] Usar BlueprintNativeEvent cuando se necesite implementación base en C++ y extensión en Blueprint.
[ ] Mantener referencias débiles o suaves cuando convenga evitar cargas innecesarias.
[ ] Evitar referencias circulares.
[ ] Documentar dependencias.
[ ] No cambiar una clase base sin revisar sus Blueprints hijos.
[ ] Compilar completamente después de cambios estructurales.
[ ] No depender exclusivamente de Hot Reload para cambios de UCLASS, USTRUCT o UPROPERTY.

DATA-DRIVEN DESIGN
------------------
Evitar código rígido como:

- Si es Noche 1.
- Si Brianna habló.
- Si el libro fue inspeccionado.
- Entonces generar Archivista.

En su lugar:

Evento:
Archivist.FirstAppearance

Condiciones:
- Chapter.Night01.Library.
- Clue.Library.Book.Inspected.
- Friend.Brianna.Present.

Acciones:
- Apagar luces.
- Reproducir sonido.
- Generar entidad.
- Bloquear salida.
- Iniciar objetivo.

Estas condiciones deben poder configurarse mediante:

- Data Assets.
- Data Tables.
- Gameplay Tags.
- Blueprints de evento.
- Componentes reutilizables.

Convenciones:
[ ] A para Actor.
[ ] U para UObject y ActorComponent.
[ ] F para struct.
[ ] E para enum.
[ ] I para interfaz.
[ ] b para booleanos.
[ ] Nombres completos y claros.
[ ] Categorías coherentes en UPROPERTY y UFUNCTION.
[ ] Comentarios para decisiones, no para repetir el código.
